within the Core dir, create a local html page that uses thr rustplus.js node library to do things. 
It should be modular, documented, and GUI functionality should be implemented in such a way that a gui functionality is the same as a non-gui functionality, just with an extra layer of abstraction. for example, if we want to turn on a smart switch, we should have a function that turns on a smart switch, and then the GUI should call that function when the user clicks a button.
These are mandatory requirements:
- Code must be modular, documented, and well structured, updating code does not always require a new file to be created
- The GUI should be implemented in such a way that it is easy to add new functionality without having to change the existing code too much, for example, if we want to add a new type of device, we should be able to do that without having to change the existing code too much.
- Pairing with a server should be able to be done in the gui and non-gui.
- Smart switches must be able to be added, removed, turned on, and turned off, the current state of the switch also needs to be accurately represented in the GUI.
- Smart switchs, Cameras, and other devices should be able to be grouped together in the GUI. 
    - Smart switches that are grouped together should be able to be turned on or off together via the group, the current states of the switches in the group should also be accurately represented by the total number turned off or on in the group. Switches in a group should also be able to be turned on or off individually, and the group should update accordingly.
    - Cameras in a group should be able to be viewed on a rotating cofigured timer. For example, if we have 3 cameras in a group, we should be able to configure the timer to rotate between the cameras every 5 seconds, and the GUI should update accordingly to show the current camera being viewed. The user should also be able to manually switch between cameras in the group, and the timer should pause upon user input and resume after resumed manually.
    - Smart alarms in a group should consolidate notifications that occur within a configurable time frame into a single notification, for example, if we have 3 smart alarms in a group and they all go off within 2 seconds of each other, only one notification should be sent to the user. By default all smart alarms in a group should be consolidated into a single notification, but the user should be able to configure the time frame for consolidation, disable consolidation entirely, and be able to exclude individual smart alarms from consolidation. Regardless of consolidation, all notifications from smart alarms should be logged in the backend. Consolidated is simply choosing not to show the user certain notifications.
    - Turrets in a group should be able to be viewed on a rotating configured timer, similar to cameras, but instead of showing the current camera being viewed, it should show the current turret being viewed, and the user should be able to manually switch between turrets in the group, and the timer should pause upon user input and resume after resumed manually.
    -Storage Monitors in a group should provide the totals across all of the storage monitors in the group, with the ability to view the individual storage monitors as well. There must be the ablility to define sub-totals from the storage monitors in the group. These should allow for exclusion of certain storage monitors, certain items, or certain item categories. For example, for a group we should have the grand total which shows each items total across all storage monitors in the group and sub-totals that are a subset of the grand total defined by any combination of storage monitors, included/excluded items, included/excluded item categories. Individual Itemsor categorys should also have their delta change shown. By default, tracking delta change should be disabled, but the user should be able to enable it for any item, category, group, or subtotal. Their should also be the ability to show a graph of the change over a rolling length of time (Last hour).
    - Storage monitor groups should also have the ablility to have item requirement tracking. These should allow for action to be taken based on a defined requirement. A not inclusive list of requirements could be: falling below a certain total for an item, rising above a certain total for an item, falling below a certain total for a category of items, rising above a certain total for a category of items, falling below a certain delta change for an item, rising above a certain delta change for an item, falling below a certain delta change for a category of items, rising above a certain delta change for a category of items. Actions that can be taken based on these requirements should include: sending a notification to the user, turning on or off a smart switch, turning on or off a group of smart switches, sending a team message in game. Additionally, the ability to calculate an estimated time until a requirement is met based on the an average delta change over a defined rolling length of time should be implemented. By default this should be disabled, but the user should be able to enable it for any item, category, group, or subtotal, or requirement. Additionally, the ability to do this calculation one time on demand should also be implemented. The on demand should use the delta change at the moment the request is made. It is important that delta change features must be implemented in a manner that does not require additional requests to the Rust+ server. If addtional requests are required, then they must be limited and infrequent. Failure to do so will result in being rate limited by the Rust+ server very quickly.
- An API or something similar should be implemented to allow for external scripts to interact with the backend functionality, for example, an external script should be able to turn on a smart switch, and the GUI should update accordingly to show the current state of the switch. This API should also allow for external scripts to receive notifications from the backend, for example, if a smart alarm goes off, an external script should be able to receive that notification and do something with it. This API should also allow for external scripts to receive updates from the backend, for example, if a storage monitor total changes, an external script should be able to receive that update and do something with it.
- The backend should be implemented in such a way that it can handle a large number of devices and groups without significant performance degradation.
- The implementation should allow for multiple servers to be paired with and easily switched between in the GUI, as well as the ability to have different devices and groups for each server. There should also be the ability to have a default server that is automatically shown in the GUI upon opening the application, but the user should be able to change the default server in the settings. There should be the ablility to have different users each with their own default server, devices, and groups, but this is not a mandatory requirement.
- External dependencies should not be used, if not possible, limited.

- The understanding of the codebase for use and future development by other developers or an AI agent is mandatory. A doc directory should be created and maintained with documentation on the codebase, how to use it, and how to develop it further. Two files that can be used for AI context are mandatory, the first should be created and called AI_AGENT_CONTEXT_MASTER.md, this file should be a comprehensive document that provides an overview of the codebase, its structure, and how to use it, after it's initial creation agents should not update this file. The second file should be created and called AI_AGENT_CONTEXT_UPDATE.md, this file should be additions that agents add to the context for future agents to use. Both of these files should be used together to provide context to future agents, when the second file does not agree with the Master file, agents should clarify with a human before proceeding and update the second file to resolve the disagreement. The master file should be updated by a human when necessary, but not by agents. The update file should be updated by agents when necessary, but not by humans.
